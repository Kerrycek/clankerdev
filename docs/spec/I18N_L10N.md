# Localization (l10n) and UI translations

Requirements:
- UI must support English and Czech.
- English is the default.
- If the browser preferred language is Czech (`cs`), default to Czech.
- Users can override language explicitly.
- Provide **proper Czech translation** (not machine-quality placeholders).

## Language model

- UI setting: `language: 'system' | 'en' | 'cs'`.
- `system` means:
  - if `navigator.languages` contains `cs` (or `cs-*`), use `cs`
  - else use `en`

## Translation storage

Use in-repo dictionaries:
- `src/i18n/en.ts`
- `src/i18n/cs.ts`

Each exports an object of string keys:

```ts
export const en = {
  'nav.vpses': 'VPSes',
  'actions.start': 'Start',
  // ...
} as const;
```

## Translation API

Provide a small typed translation hook:
- `t(key, params?)`
- interpolation with `{name}` placeholders

Implementation (current):
- Provider: `src/app/i18n.tsx` (`I18nProvider`)
- Hook: `useI18n()`
- The provider derives `lang` from `ui.settings.language` and the browser locale.
- It also exposes `preferredLanguageCodes` (`['cs','en']` or `['en','cs']`) which is used
  when picking API-provided translated fields (outages, etc.).

Rules:
- Avoid string concatenation in UI.
- Prefer full sentences with placeholders.

### Pluralization

Czech pluralization must be correct for key user-facing counts (Members, VPS count, ...).

Implementation approach (v1):
- Provide a `tp(baseKey, count, params)` helper.
- It uses `Intl.PluralRules(locale)` to pick a category.
- The final lookup key is `${baseKey}.${category}`.

Example keys:

```ts
// EN
"members.count.one": "{count} member",
"members.count.other": "{count} members",

// CS
"members.count.one": "{count} člen",
"members.count.few": "{count} členové",
"members.count.other": "{count} členů",
```

### Key namespaces

We use stable key namespaces so strings remain consistent across modules.

Required namespaces (v1):
- `common.*`
- `nav.*`
- `state.*`
- `action.*` (action labels by action id)
- `confirm.*` (confirm dialogs)
- `gate.*` (disabled reasons; see `ACTION_GATING_ENGINE.md`)
- `error.*` (errors; see `ERRORS_AND_EMPTY_STATES.md`)
- `empty.*` (empty states; see `ERRORS_AND_EMPTY_STATES.md`)
- `toast.*` (notifications; see `NOTIFICATION_COPY_TEMPLATES.md`)

Rule: keys must be treated as a public API for the UI.
Renaming keys requires updating both EN and CS and adding a changelog note.

Rule: avoid “hacky” pluralization like "Počet členů: {count}" unless necessary.

## Date/number formatting

- Use `Intl.DateTimeFormat`, `Intl.NumberFormat` with the selected locale.
- Use 24h format for Czech.

## API-provided translated content

Some API objects contain multiple language fields, e.g. `en_summary`, `cs_summary`,
`en_description`, `cs_description` (outages, help boxes, etc.).

Rules:

- Supported UI languages are **only** `en` and `cs`.
- Use the selected UI language as the **first choice** when picking these fields.
- Then fall back to the other supported UI language.

We intentionally do not attempt to select translations for other browser languages
(`de`, `fr`, ...) because the UI does not ship those locales.

Implementation note:
- The existing helper `pickTranslation(obj, field, preferredLanguageCodes)` should be
  fed by a `preferredLanguageCodes` list derived from UI settings, **not only** from
  `navigator.languages`.
- For API objects that may expose localized variants under several field names
  (`cs_message`, `en_body`, `cs_text`, ...), use `pickLocalizedFieldFrom()`.

## API enum/value labels

The API exposes many selectable values in forms and then returns those same
values for display. Do not render raw internal values such as
`planned_outage`, `unplanned_outage` or `publish` directly in user-facing UI.

Rules:
- If the API returns localized labels, prefer the API-provided label.
- Otherwise translate known values through `api.*` keys in the UI dictionaries.
- Unknown future values must be humanized (`new_value` → `New value`), not
  displayed as raw snake_case.

Current shared helper:
- `src/lib/apiValues.ts`

Covered examples:
- `api.outage.type.planned_outage` → CS `Odstávka`, EN `Planned outage`
- `api.outage.type.unplanned_outage` → CS `Výpadek`, EN `Unplanned outage`
- `api.security_advisory.state.publish` → CS `Zveřejnit`, EN `Publish`
- `api.security_advisory.state.mitigated` → CS `Ošetřeno`, EN `Mitigated`

Transaction chains, transactions and affected object labels are localized by
the API. Prefer API labels (`label`, localized object labels, etc.) over
WebUI-local hand-written names.


## Czech quality bar

### Tone

Prefer an *impersonal / neutral* tone:
- avoids deciding between formal and informal address
- matches common admin/UIs well

Examples:
- ✅ “Zastavit VPS”
- ✅ “Nastavit heslo roota”
- ❌ “Zastavte si VPS”
- ❌ “Zastav svůj server”

### Consistency

- Use consistent terms across the whole UI.
- Avoid switching between synonyms unless it improves basic comprehension.

## Glossary (locked terminology for v1)

This glossary is intentionally small, but the terms here are **locked** for v1 so the UI
does not drift between synonyms.

General:
- Dashboard → **Přehled**
- Account / Profile → **Účet**
- Preferences / Settings → **Nastavení** (or **Předvolby** only if context needs it; default is Nastavení)

Compute/storage:
- VPS / VPSes → **VPS** (do not translate)
- Dataset(s) → **Dataset / Datasety**
- Storage (Basic label for datasets) → **Úložiště**
- Snapshot(s) → **Snapshot / Snapshoty**
  - In Basic help text, it is allowed to add: “Snapshot = snímek stavu dat”.

Networking/DNS:
- DNS zone → **DNS zóna**
- Zone(s) → **Zóna / Zóny**
- Record(s) → **Záznam / Záznamy**
- TTL → **TTL** (do not translate)

Cluster/public status:
- Status (public) → **Stav služby**
- Node(s) → **Node / Nody**
  - Rationale: matches common technical usage and avoids awkward inflections.
- Outages → **Výpadky**
- Planned outage → **Odstávka** (never “plánovaný výpadek”)
- Unplanned outage → **Výpadek**
- News → **Novinky**
- Members → **Členové**
- IPv4 left → **Volné IPv4** (or **Zbývá IPv4**; pick **Volné IPv4** for v1)

Security:
- Security advisory → **Bezpečnostní upozornění**
- Mitigated → **Ošetřeno**
- SSH public key → **SSH veřejný klíč**
- Session(s) → **Relace** (user sessions/tokens)
- Token → **Token** (acceptable loanword)

Account:
- Login/account handle → **Přezdívka** (not “přihlašovací jméno”)

Tasks/operations:
- Tasks drawer → **Úlohy**
- Recent actions → **Nedávné akce** (Basic-friendly label is allowed)
- Transaction(s) → **Transakce**
- Action state(s) → **Stavy akcí**

States:
- Running → **Běží**
- Stopped → **Zastaveno** (locked for v1)
- Suspended → **Pozastaveno**
- Deleted → **Smazáno**
- Busy → **Probíhá operace**
- Maintenance → **Údržba**
- Unknown → **Neznámý stav**

Actions:
- Start → **Spustit**
- Stop → **Zastavit**
- Restart → **Restartovat**
- Console → **Konzole**
- Copy → **Kopírovat**
- Apply → **Použít změny**
- Save → **Uložit**
- Delete → **Smazat** (use **Odstranit** only if we intentionally distinguish soft-delete vs remove)

Notes:
- Prefer short nouns in navigation (“DNS”, “VPS”) and clearer phrases in help text.
- Avoid mixing “Úlohy” and “Akce” on the same screen; choose one label per surface.


## Coverage strategy

- Start with nav + common actions + shared components.
- Then cover module-specific copy.

Add a test that verifies:
- every key in `en` exists in `cs`
- no extra keys in `cs` without `en`

## Acceptance criteria

- Fresh user with browser `cs` sees Czech UI.
- Fresh user with browser non-`cs` sees English UI.
- Language choice persists in UI settings.
- No mixed-language UI (labels consistently use selected language).

## Hardcoded-string policy

We keep i18n discipline with a lightweight audit:

- `npm run audit:ui-strings` → writes `work/audits/ui_strings.md`
- `npm run audit:ui-strings:check` → fails if any likely user-facing hardcoded strings are found

The audit prefers a TypeScript AST walk when `typescript` is available, and falls back to conservative regex heuristics otherwise.

Ignore directives (use sparingly):
- `// i18n-ignore-file`
- `// i18n-ignore`
- `// i18n-ignore-next-line`

Rationale: developer-only pages (e.g. Design Sandbox) may include intentional sample copy.
